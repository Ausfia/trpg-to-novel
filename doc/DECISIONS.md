## ADR-001：四阶段线性流水线

### 决策

将整个系统设计为 **W0 → W1 → W2 → W3** 的**单向线性流水线**，不采用 Agent 编排、循环反馈、多 Agent 协作等复杂模式。

### 背景

将 TRPG 日志转为小说是一个复杂任务，理论上可以有多种架构：
1. 单一巨型 prompt（端到端）
2. 多 Agent 协作（如"作家 Agent + 编辑 Agent + 评审 Agent"互相反馈）
3. 自反思循环（生成 → 评估 → 改进）
4. 线性流水线

### 考虑过的替代方案

**方案 A：端到端单 prompt**
- ❌ 输入太大（多个 session 的日志可能数十万 token）
- ❌ 无法精细控制各阶段
- ❌ 无法中间介入修改

**方案 B：多 Agent 协作**
- ❌ Agent 间通信成本高、容易陷入无效辩论
- ❌ 难以调试（出问题时不知道是哪个 Agent 的锅）
- ❌ 实际项目中"作家+编辑"模式被反复验证为**收益小于复杂度**
- ❌ 真正的"编辑"工作应该由人来做

**方案 C：自反思循环**
- ❌ 自反思在创作类任务上效果差（LLM 难以客观评价自己的输出）
- ❌ 容易陷入"修改 → 引入新问题 → 再修改"的恶性循环
- ❌ Token 消耗成倍增加

**方案 D：线性流水线（选定）**
- ✅ 每个阶段职责单一、易于调试
- ✅ 中间产物可以人工修改后重新进入流水线
- ✅ 失败时容易定位问题
- ✅ Token 消耗可控

### 理由

**复杂的决策应该让人来做，AI 负责执行。**

跑团转小说的核心难点不是"AI 不会写"，而是"AI 不知道你想要什么"。让 AI 自我循环不会让它更懂你，但**让你能在中间介入**会。

线性流水线把"决策点"暴露给人，把"执行"留给 AI，是当前 LLM 能力下最稳妥的架构。

### 后果

- ✅ 系统更简单、更可控
- ⚠️ 用户需要在每个阶段做 review，不是真正的"一键生成"
- ⚠️ 某些跨阶段优化（如 W3 发现 W2 大纲有问题需要回改）需要手动重跑

## ADR-002：中间产物 JSON 持久化

### 决策

每个阶段（W0、W1、W2、W3 的子产物）的输出都**强制持久化为 JSON 文件**（W3 最终产物是 Markdown）。阶段之间**只通过文件传递**，不在内存中共享对象。

### 考虑过的替代方案

**方案 A：纯内存传递**
- 把 W0 的输出对象直接传给 W1，不写文件
- ❌ 无法中间介入
- ❌ 无法断点续传
- ❌ 调试困难（出错时不知道当时的输入是什么）

**方案 B：用 SQLite / 数据库**
- ❌ 增加依赖和复杂度
- ❌ JSON 文件已经够用，且更易于人工编辑
- ❌ 失去"用文本编辑器查看/修改"的便利

**方案 C：JSON 文件（选定）**
- ✅ 人类可读、可编辑
- ✅ 天然支持 git diff、版本控制
- ✅ 任何工具都能处理
- ✅ 阶段间天然解耦

### 理由

**结构化中间产物是这个项目的核心资产**，比生成的小说本身更有价值——因为它是可复用的素材库。

JSON 文件让用户能在任何阶段：
1. 用 VSCode / 编辑器直接修改
2. 用 git 追踪变更
3. 用脚本批量处理
4. 用其他工具二次开发

### 后果

- ✅ 极高的灵活性和透明度
- ⚠️ 大文件的读写有少量开销（可忽略）
- ⚠️ 用户需要会基本的 JSON 编辑（但 JSON 是程序员通用知识，不算门槛）

## ADR-003：人在回路

### 决策

每个阶段结束后是**强制的 review 点**。系统设计不追求"一键生成完整小说"，而追求"每个阶段产出高质量中间产物，让用户能确认/修改后进入下一步"。

### 背景

存在两种产品哲学：
- **全自动**：用户提供输入 → 系统直接产出成品
- **协作式**：用户与系统多次交互、迭代

### 考虑过的替代方案

**方案 A：全自动模式**
- 提供 `--all` 选项一口气跑完全部阶段
- ❌ 当前 LLM 能力下，无法保证每个阶段都对
- ❌ 错误会向下传播放大
- ❌ 最终发现问题时已经消耗大量 token

**方案 B：协作式（选定）**
- 默认行为：用户每跑完一个阶段就 review
- 提供 `--all` 但加警告
- ✅ 错误能早期发现、早期修正
- ✅ 用户对最终产物有更强的掌控感

### 理由

跑团是**用户的回忆和创作**，不是"待处理的数据"。用户对最终小说应该有强烈的所有权感。

人在回路不是"AI 能力不足的妥协"，而是**对用户创作权的尊重**。

### 后果

- ✅ 最终产物质量更高
- ✅ 用户更愿意分享、传播（因为是"自己的作品"）
- ⚠️ 完整流程时间更长（但通常用户能接受）

## ADR-004：场景作为最小叙事单元

### 决策

整个系统的核心数据结构是 **Scene（场景）**。一切分析、规划、生成都围绕场景展开。

### 背景

可选的最小单元有：
- 段落 / 句子
- 事件 / 动作
- 章节
- **场景**（选定）

### 考虑过的替代方案

**方案 A：以"事件"为单位**
- 把日志拆成"动作 1 → 动作 2 → ..."
- ❌ 颗粒度过细，丢失情境
- ❌ 难以做风格化处理

**方案 B：以"章节"为单位**
- 直接让 LLM 处理整章
- ❌ 上下文过大，质量下降
- ❌ 难以细粒度控制

**方案 C：以"场景"为单位（选定）**
- ✅ 符合戏剧/小说的天然结构（剧本就是按场景写的）
- ✅ 颗粒度适中（一个场景几百到几千字）
- ✅ 时空连续 + 焦点连续，是天然的语义单元
- ✅ 便于灵活组装（同样的场景池可以组装出不同的章节结构）

### 理由

**场景是叙事的原子**。莎士比亚、电影剧本、小说写作教程，全都按场景思考。

把场景作为最小单元让系统的每个阶段都有清晰的对象：
- W1 提取场景
- W2 把场景组装为章节
- W3 把场景渲染为文字

### 后果

- ✅ 数据模型清晰
- ✅ 各阶段职责明确
- ⚠️ "什么是一个场景"需要在 W1 中精确定义（已在 DESIGN.md 4.3 节说明）

## ADR-005：双层生成策略

### 决策

W3 正文生成采用 **"逐场景生成 → 章节缝合"** 的双层策略，而非"直接生成整章"。

### 考虑过的替代方案

**方案 A：直接生成整章**
- 给 LLM 整章的大纲，让它一次生成
- ❌ 上下文过大，模型注意力分散
- ❌ 难以中途修改（一处不满意要重生成整章）
- ❌ 不同场景的细节质量不均

**方案 B：直接生成全书**
- ❌ 显然不现实

**方案 C：双层生成（选定）**
- 第一层：逐场景生成 SceneDraft
- 第二层：章节缝合（调整过渡 + 添加章节头）
- ✅ 每个场景的生成都聚焦、精确
- ✅ 失败时只需重生成单个场景
- ✅ 用户可以对特定场景做精修

### 理由

类似"分而治之"。**局部精度**和**全局连贯**是两个不同的优化目标，分两步处理比一步到位效果更好。

### 后果

- ✅ 单场景质量更高
- ✅ 断点续传粒度更细
- ⚠️ 场景间衔接需要额外处理（已通过"传递前一场景结尾"机制解决）

## ADR-006：风格配方两层叠加

### 决策

风格控制分两层：
1. **全局风格配方**（`config/style_recipe.yaml`）：定义整本书的基础风格
2. **章节级风格指令**（在 `novel_outline.json` 中）：根据章节内容微调

最终生成时，两层指令**叠加合成**为完整的 prompt。

### 考虑过的替代方案

**方案 A：只有全局风格**
- ❌ 战斗章节和日常章节用同样的指令，效果生硬

**方案 B：每章完全自由风格**
- ❌ 全书风格不统一

**方案 C：两层叠加（选定）**
- ✅ 保证全书风格基线一致
- ✅ 允许章节级微调
- ✅ 用户可以在 W2 阶段精细调整每章风格

### 理由

写作风格不是单一维度的。"少年漫风格"是基线，但"战斗章""日常章""转折章"应该有不同的处理。

两层叠加给了用户两个层次的控制点：粗调（修改配方）+ 细调（修改单章指令）。

### 后果

- ✅ 灵活性高
- ⚠️ 用户需要理解"全局风格 + 章节微调"的关系（已在 DESIGN.md 5.3.5 节说明）

## ADR-007：单人剧情四种处理

### 决策

单人剧情有四种处理方式，由 W2 阶段决定：
1. **独立成章**
2. **闪回嵌入**
3. **拆分嵌入**
4. **降级为参考资料**

### 背景

单人剧情是 TRPG 跑团的独特元素——某个 PC 与 KP 单独跑的剧情，不在主线中。这些剧情的特点是：
- 主线中其他 PC 不知情
- 时间上可能发生在主线之前/之间
- 篇幅差异极大（从几百字的回忆片段到完整的 session）
- 重要性差异极大（从角色心境的关键转折到设定补充）

### 考虑过的替代方案

**方案 A：一律独立成章**
- ❌ 短的单人剧情独立成章太单薄
- ❌ 打断主线节奏

**方案 B：一律插入主线**
- ❌ 长的单人剧情塞进闪回会破坏主线

**方案 C：忽略单人剧情**
- ❌ 丢失重要素材

**方案 D：四种处理方式（选定）**
- ✅ 根据剧情特点灵活选择
- ✅ 决策点暴露给用户，可在 W2 review 时调整

### 理由

跑团的核心魅力之一就是"每个 PC 都有自己的故事"。**单人剧情处理是否得当，直接决定小说是否有"群像感"**。

四种方式覆盖了实际场景：
- 长且独立的弧线 → 独立成章
- 短且有共鸣的回忆 → 闪回
- 长但多段的剧情 → 拆分嵌入
- 纯设定补充 → 降级为参考

### 后果

- ✅ 处理灵活、贴合实际
- ⚠️ W2 阶段对单人剧情的判断需要较强的 LLM 能力（已在配置中允许为 W2 使用最强模型）

## ADR-008：伏笔统一规划

### 决策

伏笔的"埋设 - 强化 - 回收"由 **W2 阶段统一规划**，而不是依赖日志中自然存在的伏笔。如果日志中没有自然的埋设点，W2 会**主动设计补写的伏笔片段**，作为章节中的小场景插入。

### 背景

跑团日志中的伏笔状况通常是：
- KP 埋的伏笔可能玩家没接住（=没有"强化"和"回收"）
- 玩家发现的线索可能 KP 没想到（=意外伏笔）
- 人物秘密（人物卡上写明的）可能完全没在主线中体现

### 考虑过的替代方案

**方案 A：只用日志中已有的伏笔**
- ❌ 大量人物秘密会被埋没
- ❌ 小说结构松散

**方案 B：完全由 AI 自由编造**
- ❌ 可能引入与原团本矛盾的内容

**方案 C：W2 统一规划 + 主动补写（选定）**
- ✅ 保证关键秘密都有"埋设 → 回收"的弧线
- ✅ 补写的内容受人物卡和原日志约束，不会偏离
- ✅ 用户可在 W2 review 时调整伏笔分布

### 理由

**小说不是日志的转写，而是日志的再创作**。伏笔的精心安排是小说与日志的核心区别之一。

把伏笔规划集中在 W2，让它成为"作家级的决策"，比起在 W3 写到一半"临时加点暗示"要可控得多。

### 后果

- ✅ 小说结构更完整、有戏剧张力
- ✅ 人物秘密能得到展现
- ⚠️ 补写的内容可能与原日志气质不符（通过"人物卡 + 日志"双重约束缓解）

## ADR-009：使用 LiteLLM

### 决策

LLM 调用统一通过 **LiteLLM** 库，而不是直接使用 OpenAI / Anthropic / 各家 SDK。

### 考虑过的替代方案

**方案 A：直接使用 OpenAI SDK**
- ✅ 简单
- ❌ 锁定单一服务商
- ❌ 换服务商要改代码

**方案 B：自己封装一层**
- ❌ 重复造轮子
- ❌ 维护成本高（每加一个服务商都要写适配）

**方案 C：使用 LangChain**
- ❌ 过重、过度抽象
- ❌ 我们用不到 LangChain 的 chain / agent / memory 等功能
- ❌ LangChain 的频繁 API 变动会带来维护负担

**方案 D：使用 LiteLLM（选定）**
- ✅ 轻量，只做一件事：屏蔽不同 LLM 服务商的差异
- ✅ 支持 100+ 模型，包括本地服务（Ollama / vLLM / LM Studio）
- ✅ API 稳定、社区活跃
- ✅ 完美适配我们"用户自由配置每个环节的模型"的需求

### 理由

LiteLLM 是当前生态里**最匹配"多服务商支持"需求**的库。它不试图做更多事，只做适配，这正是我们需要的。

### 后果

- ✅ 用户可以为每个环节自由选择服务商
- ✅ 切换服务商零代码修改
- ⚠️ 偶尔某些新模型/新功能 LiteLLM 跟进慢（可通过 base_url 直连绕过）

## ADR-010：Phase 1 不做 WebUI

### 决策

Phase 1（当前）只实现 CLI。WebUI 是 Phase 2 目标，但当前架构必须保证**未来可以低成本叠加 WebUI**。

### 考虑过的替代方案

**方案 A：从一开始就做 WebUI（Gradio）**
- ✅ 用户体验更好
- ❌ 工作量翻倍
- ❌ 在核心流水线未验证时做 UI 是过早优化
- ❌ UI 调试会分散对 prompt 调优的注意力

**方案 B：永不做 WebUI，纯 CLI**
- ❌ 限制了未来的可能性（如分享给非技术朋友用）

**方案 C：Phase 1 CLI + 架构预留（选定）**
- ✅ MVP 先验证核心价值
- ✅ 架构上零成本预留（用纯函数 + 回调 + 文件持久化）
- ✅ 验证有价值后再做 WebUI，约 2-3 天可叠加

### 理由

**WebUI 是放大器，不是引擎**。如果核心流水线本身产出垃圾，WebUI 只会让你更快地发现这是垃圾。

先把引擎做好，再做放大器。

### 后果

- ✅ Phase 1 工作量可控
- ✅ 早期就建立了"业务与界面解耦"的好习惯
- ⚠️ 当前用户必须接受 CLI（但目标用户就是项目开发者本人，可接受）

## ADR-011：模型配置扁平结构

### 决策

`config/models.yaml` 采用 **default + 覆盖** 的扁平结构，**不引入 "profile" 等抽象概念**。

### 背景

设计模型配置时，考虑过两种结构：

**结构 A：profile + assignments（两层抽象）**
```yaml
profiles:
  gpt4o_official:
    model: gpt-4o
    api_key_env: OPENAI_API_KEY
  claude:
    model: claude-sonnet-4-5
    api_key_env: ANTHROPIC_API_KEY
assignments:
  w2_outline.chapter_planning: claude
```

**结构 B：default + 覆盖（扁平）**
```yaml
default:
  model: gpt-4o
  api_key_env: OPENAI_API_KEY
workflows:
  w2_outline:
    chapter_planning:
      model: claude-sonnet-4-5
      api_key_env: ANTHROPIC_API_KEY
```

### 考虑过的替代方案

**方案 A：profile + assignments**
- ✅ 一个服务商配置可以被多个环节复用
- ❌ 多一层抽象，用户要学"profile 是什么"
- ❌ 对 4 个工作流 × 几个步骤的规模，复用收益小

**方案 B：扁平结构（选定）**
- ✅ 一看就懂，无需学习概念
- ✅ 直接修改某个环节，不需要全局思考
- ⚠️ 多个环节用同一服务商时配置会重复（已通过 default 继承缓解）

### 理由

**对用户友好优先于代码优雅。**

这个项目的用户是"想跑团变小说的爱好者"，不是"想学习配置 DSL 的工程师"。让用户直接看到每个环节用什么模型，比让他理解 profile 抽象更重要。

### 后果

- ✅ 配置文件易于理解
- ✅ 新增/修改某个环节的配置直接、无副作用
- ⚠️ 几个环节用同一非默认服务商时，会重复写 `api_key_env` 和 `base_url`（可接受）

## ADR-012：Prompt 用 Jinja2

### 决策

所有 prompt 用 **Jinja2 模板**写在 `prompts/*.j2` 文件中，与 Python 代码完全分离。

### 考虑过的替代方案

**方案 A：Prompt 写在 Python 字符串里**
- ❌ 难以编辑（长字符串 + 转义）
- ❌ 难以版本控制（diff 不清晰）
- ❌ 改 prompt 要改代码、要重新测试

**方案 B：Prompt 写在 .txt 文件里，用 f-string 注入**
- ✅ 比 A 好
- ❌ 复杂逻辑（条件、循环）难以表达

**方案 C：Jinja2 模板（选定）**
- ✅ 支持变量、条件、循环、include
- ✅ Python 生态主流模板引擎，无学习成本
- ✅ 可以拆分公共片段（如人物卡格式化）
- ✅ 改 prompt 不需要改代码、不需要测试

### 理由

Prompt 是这个项目的**核心智力资产**，应该像代码一样被精心管理：
- 独立文件存放
- 版本控制
- 易于 review 和 diff
- 可以做"prompt 实验"（备份旧版本，尝试新版本）

Jinja2 是 Python 生态中表达力和易用性最佳的模板引擎。

### 后果

- ✅ Prompt 工程化、可维护
- ✅ 非程序员也能改 prompt（只要懂 Jinja2 语法基础）
- ⚠️ 引入一个依赖（jinja2，但 Python 生态中几乎是标配）

## ADR-013：使用 Pydantic + Instructor

### 决策

所有需要结构化输出的 LLM 调用，使用 **Pydantic 模型定义 schema** + **Instructor 库强制 LLM 返回结构化数据**。

### 考虑过的替代方案

**方案 A：让 LLM 自由返回 JSON 字符串，自己解析**
- ❌ LLM 经常返回格式错误（多个 ```、多余逗号、字段名错误）
- ❌ 需要自己写大量解析和容错代码

**方案 B：用 OpenAI 的 `response_format: json_object`**
- ✅ 保证返回合法 JSON
- ❌ 不保证字段符合 schema
- ❌ 只支持 OpenAI 系列模型

**方案 C：Pydantic + Instructor（选定）**
- ✅ 用 Pydantic 定义 schema，类型安全
- ✅ Instructor 自动做：注入 schema 到 prompt + 解析 + 验证 + 失败重试
- ✅ 支持所有主流 LLM（OpenAI、Anthropic、Gemini 等）
- ✅ 验证失败时自动在 prompt 中追加错误信息让 LLM 修正

### 理由

LLM 的结构化输出是这个项目的核心需求（每个阶段都有结构化产出）。手写解析和验证代码是徒劳的——Instructor 已经把这件事做到了极致。

### 后果

- ✅ 大幅减少格式解析代码
- ✅ 类型安全（IDE 自动补全、mypy 检查）
- ✅ Pydantic 模型直接序列化为 JSON，无需手动处理
- ⚠️ Pydantic schema 必须设计得清晰（这是好事，强迫思考清楚字段）

## ADR-014：日志切分两阶段

### 决策

日志切分分两个阶段：
- **W0**：按 token 数粗切（4000 tokens / 块，重叠 200）
- **W1**：在粗切基础上做**语义级场景切分**（基于滑动窗口）

### 考虑过的替代方案

**方案 A：W0 一次切到语义场景**
- ❌ W0 是预处理阶段，不应该消耗大量 LLM 调用
- ❌ 语义切分需要理解叙事，是 W1 的职责

**方案 B：W1 直接处理整个日志文件**
- ❌ 上下文过大，LLM 注意力分散
- ❌ 长日志会超出 token 限制

**方案 C：两阶段切分（选定）**
- W0 物理切分：快速、便宜
- W1 语义切分：精确、贵但只对粗块做
- ✅ 各阶段职责清晰
- ✅ 性能和质量都可控

### 理由

切分粒度的不同应该匹配阶段的目标：
- W0 只需要"让 W1 能处理"
- W1 才需要"语义正确"

强行让一个阶段同时做两件事会丢失清晰度。

### 后果

- ✅ 各阶段职责清晰
- ⚠️ W1 需要处理"跨块场景"的情况（已通过滑动窗口 + 上文传递解决）

## ADR-015：不做 RAG

### 决策

**不引入 RAG / 向量数据库 / Embedding 检索**。所有上下文通过"精心组装的 prompt"传递。

### 考虑过的替代方案

**方案 A：用 RAG 检索相关上下文**
- 比如 W3 写某个场景时，检索人物卡、相关历史场景
- ❌ 增加大量基础设施（向量库、embedding API）
- ❌ 检索结果质量难以保证
- ❌ 我们的"相关上下文"在结构上已经明确（人物卡、场景池），不需要检索

**方案 B：直接组装 prompt（选定）**
- W3 写场景时，明确知道要用哪些信息：
  - 本场景的结构化数据
  - 本章主题
  - 涉及角色的人物卡
  - 前一场景的结尾
- ✅ 信息组装精确、可控
- ✅ 无额外基础设施

### 理由

**RAG 解决的是"我不知道哪些信息相关"的问题**。但在这个项目里，每个生成步骤的相关信息都是**结构上明确的**（人物卡、场景、章节），不需要检索。

引入 RAG 是为"以备万一"，但实际产生的复杂度远超收益。

### 后果

- ✅ 系统更简单
- ✅ 上下文组装可控、可追溯
- ⚠️ 如果未来场景数量极大（如几百章的连载），可能需要重新评估

## ADR-016：CLI 优先解耦

### 决策

业务逻辑（workflow / orchestrator）与界面层（CLI）**完全解耦**。CLI 只做：
1. 命令解析
2. 调用 workflow（传入配置和回调）
3. 接收回调并显示进度

### 理由

这是为 ADR-010（Phase 2 加 WebUI）做的准备。所有业务函数都是纯函数，进度通过回调上报，状态持久化在文件，这样将来加 WebUI 时：
- 不需要重构业务代码
- 只需要写一层新的"界面壳"

### 关键约束（实现时必须遵守）

1. **workflow 函数不直接调用 `print` / `rich.print`**
2. **进度通过 `progress_callback` 上报**
3. **没有 CLI 特有的全局状态影响业务逻辑**
4. **任何"运行进度"通过 `workspace/` 文件持久化**

### 后果

- ✅ 未来扩展 WebUI 成本低
- ✅ 业务逻辑更易于单元测试
- ⚠️ 初期需要克制"直接 print 一下"的冲动
